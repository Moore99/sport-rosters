# Lineups & Position Assignments

## What It Does
Lets team admins build event lineups — either manually (drag-and-drop) or via an auto-generator that uses player rankings and position preferences. Supports single-roster events and multi-sub-team events (e.g., split into two balanced squads). Dragon Boating has a dedicated boat seating screen with weight-balance logic.

## User-Facing Behaviour
| Action | Who |
|--------|-----|
| Build / edit lineup manually | Team admin |
| Auto-generate lineup (rewarded ad gates this) | Team admin |
| View finalised lineup | Team member |
| Set own position preferences (per team) | Player or admin |
| Save lineup template (reuse across events) | Team admin |
| Export lineup as PDF | Team admin |
| View boat seating layout (Dragon Boating) | Team admin |
| Auto-balance boat seating by weight (rewarded ad gates this) | Team admin |

## Firestore Rules
- Team admins write lineups; team members read.
- Players read and write their own preferences; admins read/write all preferences.
- Lineup readable if doc exists and user is a team member. Null resource (no lineup yet) handled by guard.

## Sport Configuration
Positions and categories are hardcoded in `app_config.dart`. 23 sports defined. Each sport has:
- A flat position list (e.g., Ice Hockey: Centre, Left Wing, Right Wing, Left Defence, Right Defence, Goalie)
- Named categories (e.g., "Any Forward", "Any Defence") for preference matching

Helper: `positionMatchesPreference(position, preference, sport)` — returns true if a position satisfies a preference (exact match, category match, or wildcard "Any").

### Dragon Boating Specifics
- 22 positions: Row 1–10 Left/Right + Drummer + Steersperson
- Boat config: 1–4 boats, 8–22 paddler seats (must be even), drummer toggle
- Categories: Left Side, Right Side, Front (Rows 1–3), Middle (Rows 4–7), Back (Rows 8–10), Any Paddler
- Boat seating screen balances total weight per boat, with configurable seat layout

## Auto-Lineup Generator
Respects, in priority order:
1. Exclude `coachOnly` admins (from `adminRoles` subcollection)
2. Only include players who RSVPed 'yes' (from availability docs)
3. Match player preferences to positions (`positionMatchesPreference`)
4. Prefer higher-ranked players (from `rankings` subcollection — admin-only read)
5. Fill remaining positions with unpreferred available players

Setting `autoGenerated = true` on save triggers a rewarded ad gate — the admin must watch a rewarded ad before the auto-generated lineup is committed.

## Sub-Teams
When `event.numSubTeams > 1`, the lineup screen shows tabs — one per sub-team. The generator distributes available players across sub-teams using a snake-draft approach (alternating picks) to balance skill levels. Goalie can be pre-assigned before the draft runs.

## Key Decisions
- **Doc ID = eventId** for lineups — one lineup per event. If an event needs to store multiple lineup variants, the model would need to change.
- **`subTeams` vs `assignments`** — mutually exclusive in practice. `assignments` is used for single-roster events; `subTeams` is used when `numSubTeams > 1`. The model stores both fields, but only one is populated.
- **Rewarded ad gates auto-generate** — this is a monetisation touchpoint. Users who purchased "Remove Ads" skip the ad but still trigger the same flow (ad is bypassed server-side by checking `adFree`).
- **Template ID is composite** — `{teamId}_{sport}` — so each team gets one template per sport. Overwriting is intentional (last save wins). Not designed for multiple saved templates per team.
- **Preferences are per-team**, not global. A player on two teams can have different preferences for each.

## Future / Deferred
- **Multiple lineup templates** — current model stores one template per team+sport. Named/versioned templates would support multiple lineup strategies.
- **Lineup history** — no versioning. Once a lineup is saved, the previous version is overwritten. An audit log of changes would help coaches review decisions.
- **Player-visible finalised lineup** — currently lineups are readable by all team members, but there is no prominent player-facing screen to see "where am I playing tonight". A simplified view card on the event detail screen would close this gap.
- **Drag-and-drop on tablet/web** — manual lineup builder works but the UX is optimised for phone. Tablet layout and web drag-and-drop are not explicitly handled.
- **PDF/CSV export** — implemented (Phase 3), but the spec details (layout, fields included) are not captured here.
- **Auto-balance boat seating** — implemented (Phase 4) with weight data. The exact balancing algorithm (min weight delta across boats) is in `BoatSeatingScreen` logic and not extracted to a testable function.
